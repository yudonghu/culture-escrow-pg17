## 2) 目标状态（To-Be）

达到“可生产运行”的最低标准：
1. 可控：权限、审计、风险边界清晰
2. 可稳：高峰不崩、失败可重试、错误可定位
3. 可运维：有监控、有告警、有回滚
4. 可治理：数据留存策略、版本管理、变更可追溯

---

## 3) 分阶段计划（建议 4 周）

## Phase A（Week 1）稳定性与可观测性

### A1. 错误体系标准化
- 统一 error code（400/413/422/500 子码）
- 统一返回格式：`message / error_code / hint / request_id`

### A2. 日志与追踪
- 每次任务生成 `request_id/job_id`
- API 记录关键阶段耗时（上传/处理/导出）

### A3. 健康检查增强
- `/health` 返回依赖检查状态（engine 可用性、磁盘空间简检）

**验收标准**
- 任意失败都可定位到 error code + request_id
- 关键路径耗时可见

---

## Phase B（Week 2）安全与治理

### B1. 认证与权限
- 最低可用：单租户登录（管理员口令）
- 操作级权限：运行/下载/管理分离（可先简化）

### B2. 审计日志
- 记录：谁在何时处理了哪份文件、用了哪些输入变量
- 支持按 `request_id` 查询

### B3. 文件留存策略
- 默认短期留存（如 7 天）
- 支持手动即时删除
- 日志脱敏（不明文输出敏感字段）

**验收标准**
- 能回答“谁在什么时候处理了哪一个 job”
- 可配置留存并可执行删除

---

## Phase C（Week 3）任务可靠性

### C1. 异步任务队列（轻量）
- 前端提交后返回 job_id
- 后端队列处理，避免请求超时

### C2. 重试与幂等
- 可重试失败任务（限制重试次数）
- 同一 request_id 防重复处理

### C3. 超时与取消
- 任务超时可标记失败
- 支持取消长任务（后续可选）

**验收标准**
- 高峰并发下不阻塞主线程
- 失败任务可重试并留痕

---

## Phase D（Week 4）上线准备

### D1. 环境分层
- dev / staging / prod 三环境
- 配置文件与 secrets 分离

### D2. 部署与回滚
- 版本化发布（tag）
- 一键回滚到上一个稳定版本

### D3. 监控与告警
- 指标：成功率、失败率、平均耗时、队列长度
- 告警：连续失败、响应超时、磁盘异常

**验收标准**
- 可在 staging 完整压测并通过
- 生产发布与回滚可演练

## 5) 风险与应对

1. 模板漂移导致填充偏移
- 应对：模板版本标识 + 锚点检测 + 422 明确提示

2. 依赖升级导致行为变化
- 应对：锁版本 + staging 回归测试

3. 敏感数据泄漏风险
- 应对：日志脱敏 + 最小留存 + 权限隔离

4. 并发导致性能下降
- 应对：队列化 + 限流 + 预警阈值
